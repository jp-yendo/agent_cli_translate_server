# システム仕様

## 1. 概要

Agent CLI Translate Server は、XUnity.AutoTranslator の CustomTranslate エンドポイント仕様に準拠した AI 翻訳サーバーの GUI アプリケーションである。
翻訳エンジンとして次の2系統をサポートする。同時に開始できるエンジンはアプリ全体で1つのみ。

- **Agent CLI**: ローカルにインストールされた Agent CLI (コーディングエージェントのコマンドラインツール) をプロセスとして起動して翻訳する。API Key の設定は不要で、Agent CLI 側のログイン状態を利用する。
- **API 接続**: 公式ライブラリを用いて、リモート/ローカルの API エンドポイント (Ollama / OpenAI 互換 / Anthropic 互換) へ直接リクエストして翻訳する。プロバイダーごとに Base URL・API Key・モデルを設定する。

## 2. 対応 Agent CLI

| ID | 表示名 | コマンド | 提供パッケージ | 同時起動数デフォルト |
| --- | --- | --- | --- | --- |
| claude-code | Claude Code | `claude` | `@anthropic-ai/claude-code` | 5 |
| codex | Codex CLI | `codex` | `@openai/codex` | 5 |
| grok | Grok CLI | `grok` | `@xai-official/grok` | 5 |
| opencode | opencode | `opencode` | `opencode-ai` | 1 |
| opencode-ollama | OpenCode (Ollama) | `ollama` | Ollama + opencode | 1 |

- 起動可否は PATH 上のコマンド検出 (Windows: `where` / macOS・Linux: `which`) で判定する。
- 各 Agent CLI は事前にそれぞれの手順でログイン・認証済みであることが前提。
- OpenCode (Ollama) は Ollama と OpenCode を事前にインストールし、Agent CLI 個別設定へ Ollama モデル名を保存していることが前提。

## 2-2. 対応 API プロバイダー

| ID | 表示名 | 利用ライブラリ | 既定 Base URL |
| --- | --- | --- | --- |
| ollama | Ollama | `ollama` | `http://127.0.0.1:11434` |
| openai-compatible | OpenAI Compatible API | `openai` | `https://api.openai.com/v1` |
| anthropic-compatible | Anthropic Compatible API | `@anthropic-ai/sdk` | `https://api.anthropic.com` |

- API プロバイダーは PATH 検出を行わず、常に一覧へ表示する。
- プロバイダーごとに Base URL・API Key・モデル・同時接続数を設定できる。Base URL 未設定時は既定値を使用する。
- 同時接続数 (デフォルト 1) は並行して呼び出せる API リクエスト数の上限。ローカル LLM 等では並走可能な呼び出し数が端末性能に依存するため設ける。超過分は待ち行列で処理する。
- モデル名は必須。未設定のまま開始しようとした場合はアコーディオンを開いてモデル欄へフォーカスし、入力後の保存を促す。
- API Key は認証不要なローカルサーバー等では空でもよい (OpenAI/Anthropic のライブラリへはプレースホルダーを渡して開始する)。
- 翻訳ヒントはエンジンごとではなく共通設定 (common.hintId) として全エンジンで共有する。
- 各 API 設定欄には接続テストとモデル選択を用意する。いずれも現在編集中の (未保存の) Base URL・API Key を用いて公式ライブラリのモデル一覧取得 API を呼び出す。接続テストは疎通の成否 (成功 / 失敗とエラーメッセージ) のみを表示する。モデル選択はダイアログにモデル一覧を表示し、選択した値をモデル欄へ反映する (保存は別途「保存」ボタンで行う)。モデル一覧取得には15秒のタイムアウトを設ける。

## 3. HTTP API (受信仕様)

XUnity.AutoTranslator の CustomTranslate 仕様に準拠する。

### GET /translate

`GET /translate?from={翻訳元言語}&to={翻訳先言語}&text={テキスト}`

| 条件 | ステータス | 本文 |
| --- | --- | --- |
| 翻訳成功 | 200 | 翻訳結果 (text/plain; charset=utf-8) |
| text 未指定 | 400 | 空 (キャッシュさせない) |
| 動的な値 (FPS 表示等) | 400 | 空 (キャッシュさせない) |
| 翻訳不要 (文字を含まないテキスト) | 200 | 原文をそのまま返す |
| 非対応言語コード | 400 | 空 |
| 翻訳失敗 | 500 | 空 (キャッシュさせない) |

- `from` / `to` 省略時は共通設定のフォールバック言語 (fallback-from / fallback-to) を使用する。
- 動的な値の判定: `FPS: 60` などのフレームレート表示パターン。
- 翻訳不要の判定: 空文字・空白のみ、または Unicode の Letter を1文字も含まないテキスト。

### GET /health

`ok` (200, text/plain) を返すヘルスチェックエンドポイント。

## 4. 翻訳処理

### プロンプト

プロンプト生成と応答解析は全経路で共通化し、経路差はパイプライン (プロンプトの送受信) のみとする。文面と指示内容の詳細は [プロンプト仕様](プロンプト仕様.md) を参照。本書では経路ごとの扱いのみを記す。

- プロンプトはモデル名に応じて切り替える (一般モデルは system/user 形式、Hy-MT2 系は専用形式)。1 モデルにつき 1 つ以上の「試行」を先頭から順に実行し、最初に妥当な出力を採用する。
- API 経路: system/user をロールとして送る (Anthropic は system を専用パラメータで渡す)。
- Agent CLI 経路: system/user のロールを持たないため、両者を空行区切りで 1 本に結合してプロセスへ渡す。claude-code / codex / grok / opencode はモデル名を持たないため一般モデル扱い。OpenCode (Ollama) は指定した Ollama モデル名でプロンプトが決まる。
- 共通設定の翻訳ヒント (common.hintId) が設定されている場合はプロンプトへアプリ概要を付加する。

### Agent CLI の起動方法

全ワーカーが同一プロセスを複数の翻訳依頼で再利用し、起動オーバーヘッドを通常の依頼処理経路から排除する。サーバー開始時は最低1体のプロセスについて、CLI 固有の起動完了条件を満たすまで HTTP の Listen を開始しない。最初のプロセスを起動できない場合はサーバー開始を失敗させる。

| Agent CLI | 実行方式 | 呼び出し |
| --- | --- | --- |
| claude-code | 常駐プロセス | `claude -p --input-format stream-json --output-format stream-json --verbose` を起動する。OS のプロセス起動成功後、標準入出力の JSON ストリームで同一プロセスへ複数の翻訳依頼を送る。 |
| grok | 常駐 ACP | `grok agent stdio` (ACP: JSON-RPC over stdio) を起動し、`initialize` → `session/new` の成功を起動完了条件とする。その後は同一プロセスとセッションへ `session/prompt` を繰り返し送り、`session/update` 通知の `agent_message_chunk` を連結して応答本文を得る。 |
| codex | 常駐 app-server | `codex app-server --stdio` (JSONL over stdio) を起動し、`initialize` → `initialized` → `thread/start` の成功を起動完了条件とする。その後は同一プロセスとスレッドへ `turn/start` を繰り返し送り、`item/agentMessage/delta` を連結して `turn/completed` で応答を確定する。スレッドは `ephemeral`、承認は `never`、サンドボックスは `read-only` とする。 |
| opencode | 常駐 ACP | `opencode acp --cwd <設定ディレクトリ>` (ACP: JSON-RPC over stdio) を起動し、`initialize` → `session/new` の成功を起動完了条件とする。その後は grok と同じ方式で同一プロセスとセッションを再利用する。 |
| opencode-ollama | 常駐 ACP | `ollama launch opencode --model <モデル名> --yes -- acp --cwd <設定ディレクトリ>` を起動する。Ollama が `OPENCODE_CONFIG_CONTENT` でモデル設定を OpenCode へ渡し、起動後は opencode と同じ ACP セッションを再利用する。 |

- 全 Agent CLI はコンテキスト肥大防止のため、Agent CLI ごとに設定された最大利用回数 (デフォルト 50 回) へ到達するとプロセスを再作成する。再作成後のプロセスも起動完了条件を満たしてからワーカーを再利用する。
- プロンプトはすべて標準入力経由で渡すため、cmd.exe シムの改行制限の影響を受けない。

### API プロバイダーの呼び出し方法

API プロバイダーは公式ライブラリを用いてリクエストごとに API を呼び出す。ローカルプロセスを保持しないため、最大稼働時間・最大利用回数によるプロセス再作成は行わない (ワーカーは破棄されるまで再利用可能)。同時起動数を上限とする待ち行列制御はエージェントプールが Agent CLI と共通で担う。

| プロバイダー | 呼び出し |
| --- | --- |
| ollama | `ollama` ライブラリの `chat` をストリーミングで呼び出し、チャンクを連結して応答本文を得る。タイムアウト時はクライアントの `abort()` で中断する。 |
| openai-compatible | `openai` ライブラリの `chat.completions.create` を呼び出し、`choices[0].message.content` を応答本文とする。中断は `AbortSignal` で行う。 |
| anthropic-compatible | `@anthropic-ai/sdk` の `messages.create` (`max_tokens` 指定) を呼び出し、`text` タイプのコンテンツブロックを連結して応答本文とする。中断は `AbortSignal` で行う。 |

- プロンプトはモデル名に応じて組み立てを切り替え、1 モデルにつき 1 つ以上の「試行」を先頭から順に実行して最初に妥当な出力を採用する (詳細は [プロンプト仕様](プロンプト仕様.md))。API/通信エラーは後続の試行でも解決しないため即座に中断する。
- 起動時の検証はモデル名の有無 (必須) とクライアント生成のみで、ネットワーク接続は行わない (API Key 未設定でも開始可能)。
- 1 回の翻訳実行には Agent CLI と同じ内部タイムアウト 120 秒を設ける。

- 作業ディレクトリは設定ディレクトリ (`~/.agent_cli_translate_server`) を使用する。
- Windows では npm の `.cmd` シムを経由すると改行を含む引数を渡せないため、シムから実体の JS エントリを解決し、node.exe で直接実行する。node.exe はコンソールアプリケーションのため `windowsHide` による非表示コンソールを CLI の子孫プロセスが継承し、コンソールウィンドウは表示されない。node.exe が見つからない場合のみ Electron 自身を `ELECTRON_RUN_AS_NODE=1` で利用する。シム解決に失敗した場合は cmd.exe 経由で起動する (この場合、標準入力渡しの CLI のみ動作する)。
- 1 回の翻訳実行には内部タイムアウト 120 秒を設ける (ハング検出の保険。待ちキューにはタイムアウトを設けない)。

### エージェントプール

- Listen はアプリ全体で 1 つ。開始できるエンジン (Agent CLI / API プロバイダー) も同時に 1 つのみ。
- 同時起動数の上限までワーカーを起動し、翻訳依頼を割り振る。上限は Agent CLI では同時起動数、API プロバイダーでは同時接続数の設定値を用いる。
- 上限を超えた依頼は待ち状態にする (タイムアウトなし、待ち数の制限なし)。エージェントが空き次第、先着順で再利用する。
- エージェントプロセスの終了条件は、Agent CLI ごとの最大利用回数への到達、設定された最大稼働時間の経過、サーバー停止、アプリ終了のいずれかとする。最大稼働時間 (デフォルト 300 秒) はプロセスの起動時刻から計測し、15 秒間隔で監視する。処理中に最大稼働時間へ到達した場合は、その依頼の完了後に終了する。
- 初回応答速度を維持するため、サーバー開始時に1体の起動完了を待ってから Listen を開始する。最大稼働時間の経過でエージェントが0体になった場合も、直ちに1体を起動して待機させる。
- サーバー停止時は待機中の依頼をエラーで打ち切り、全エージェントを終了する。
- アプリ終了時は終了要求を一旦保留し、稼働中の翻訳サーバーの停止と全エージェントプロセスの終了を完了させてから終了する (エージェントの終了はサーバーの close より先に同期実行し、プロセスの取り残しを防ぐ)。

## 5. 画面構成

ナビゲーションはタブではなくウィンドウのキャプション (タイトルバー) 内に組み込む。
キャプションには左側にアプリ名・バージョンに続けて翻訳サーバーの状態を表示する (停止中 / 稼働中:<エンジン表示名>)。右寄せで「サーバー / ログ / 翻訳ヒント」の切り替えボタン、共通設定 (バーガーメニューの左に歯車アイコンのみ・ラベルなし)、バーガーメニュー、ウィンドウ操作ボタンを配置する。
テーマ切替 (ライト/ダーク) と言語切替 (日本語/English) は共通設定画面で行う。
バーガーメニューのドロップダウンには画面切り替え (サーバー / ログ / 翻訳ヒント) のみを表示する。ウィンドウ幅が狭い場合はナビゲーションボタンが非表示になり、バーガーメニューから操作する。
テーマと言語の選択は設定ファイルの ui セクションに永続化される。初回起動時は OS のテーマ設定とロケールから判定した値を設定ファイルへ確定保存する。

| 画面 | 内容 |
| --- | --- |
| サーバー | 画面上部に全エンジン共通の翻訳ヒント選択欄を置き、選択を変更すると保存ボタンとは無関係に即時保存する (common.hintId)。その下に「Agent CLI」「API 接続」の2セクションに分けてアコーディオンで一覧表示。Agent CLI セクションは検出できた (コマンド起動可能な) もののみ表示し、ヘッダーにエージェント名・開始/停止ボタン、詳細に同時起動数・プロセス最大利用回数の設定と保存ボタンを表示 (OpenCode (Ollama) のみモデルも表示)。API 接続セクションは対応プロバイダーを常に表示し、詳細に Base URL・API Key・モデル・同時接続数の設定を表示。モデル欄の横にモデル選択ボタン (ダイアログで一覧から選択)、保存ボタンの左に接続テストボタンを置く。Agent CLI が1つも検出できない場合は案内メッセージを表示。 |
| ログ | 画面上部に翻訳サーバーの詳細状態を自動スクロールのチェックボックスより上に、背景色を用いずテキスト色 (稼働中は緑) のみで表示する。Agent CLI は `<エンジン表示名> (http://<host>:<port>) - 稼働中: <ワーカー数> / 処理中: <処理中数> / 待ち: <待ち数>`、API はプロセスを持たないため稼働中を省き `<エンジン表示名> (http://<host>:<port>) - 処理中: <処理中数> / 待ち: <待ち数>`。停止中は「停止中」。その下に翻訳依頼・翻訳結果などの動作状況を直近 200 件でローテーション表示。自動スクロールのチェックボックス (デフォルト ON)。ログは画面表示のみでファイルには出力しない。 |
| 共通設定 | 「表示設定」セクション (テーマ: ライト/ダーク、言語: 日本語/English。保存で即時反映) と「サーバー設定」セクション (待ち受けアドレス: デフォルト 127.0.0.1、ポート: デフォルト 4660、fallback-from: デフォルト en、fallback-to: デフォルト ja、エージェントプロセスの最大稼働時間: デフォルト 300 秒。次回のサーバー開始時に反映) に分かれ、保存ボタンは画面下部に1つ。 |
| 翻訳ヒント | アプリ概要 (サマリ) の追加・編集・削除。 |

- 開始中は他のエンジン (Agent CLI / API プロバイダー) の開始ボタンを無効化し、稼働中のボタンは「停止」に変わる。
- OpenCode (Ollama) および各 API プロバイダーの開始時に保存済みモデル名が空の場合は開始せず、対象アコーディオンを開いてモデル名入力欄へフォーカスし、入力後の保存を促す。
- 各設定セクションの保存ボタンは変更があるまで無効。保存すると再び無効化される。
- 設定変更は次回のサーバー開始時に反映される。

`~/.agent_cli_translate_server/settings.json` に保存する。詳細は [テーブル定義](テーブル定義.md) を参照。

## 7. IPC チャンネル

| チャンネル | 方向 | 内容 |
| --- | --- | --- |
| `settings:getAll` | R→M | 設定全体の取得 |
| `settings:saveCommon` | R→M | 共通設定の保存 |
| `settings:saveAgent` | R→M | Agent CLI 設定の保存 |
| `settings:saveApiProvider` | R→M | API プロバイダー設定の保存 |
| `api:testConnection` | R→M | API プロバイダーの接続テスト (モデル一覧取得の成否) |
| `api:listModels` | R→M | API プロバイダーのモデル一覧取得 |
| `hints:create` / `hints:update` / `hints:delete` | R→M | 翻訳ヒントの CRUD |
| `agents:detect` | R→M | Agent CLI の検出 |
| `server:start` / `server:stop` / `server:getStatus` | R→M | 翻訳サーバー制御 |
| `server:statusChanged` | M→R | 稼働状態の変化通知 |
| `log:getRecent` | R→M | 直近ログの取得 |
| `log:added` | M→R | ログ追加通知 |

- ログは i18n のため `key` + `params` の構造化形式でメインプロセスから送出し、レンダラー側で翻訳して表示する (`src/main/` は `src/renderer/` に依存しない)。
